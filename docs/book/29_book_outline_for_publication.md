# 29장. 출판용 책 구성안

이 장의 목적은 지금까지 작성한 분석 문서를 "읽을 수 있는 책"으로 재배치하는 것이다. 현재 `docs/book`의 파일들은 코드 분석 기준으로는 충분히 유용하지만, 그대로 출판하면 다소 기술 메모 모음처럼 보일 수 있다. 따라서 독자의 읽기 경험을 기준으로 다시 묶을 필요가 있다.

## 29.1 현재 원고의 성격

지금까지 작성된 장들은 크게 다섯 부류로 나뉜다.

1. 구조 설명 장
2. 기능 흐름 장
3. 운영/배포 장
4. 부록/참조 장
5. 비판적 분석 장

이 구성은 분석 작업에는 적합하지만, 출판용 책에서는 약간의 재배치가 필요하다. 독자는 보통 "개요 → 구조 → 핵심 흐름 → 고급 주제 → 운영 → 부록" 순으로 읽는 것이 가장 자연스럽기 때문이다.

## 29.2 추천하는 책 전체 구성

출판용으로는 전체를 다음 6부로 재구성하는 것이 가장 안정적이다.

1. 프로젝트와 구조 이해
2. 주요 SIP 흐름 따라가기
3. Transport와 TLS
4. 설정, 테스트, 운영
5. 비판적 읽기
6. 부록

각 부의 상세 장 배치는 [33_publication_toc_draft.md](/home/windmorning/projects/SIPWorks/SIPLite/docs/book/33_publication_toc_draft.md)에 별도로 정리했다. 이 파일에서는 "왜 이런 부 구성이 적절한가"에 집중하고, 실제 목차 나열은 중복을 줄이기 위해 별도 파일에 위임한다.

## 29.3 책 제목 후보

현재 원고의 성격을 반영하면 제목은 너무 학술적이기보다 "코드 분석과 실전 이해"를 드러내는 쪽이 좋다. 예를 들면 다음 후보가 가능하다.

1. `SIPLite 코드 분석`
2. `SIPLite로 배우는 SIP 서버 구조`
3. `SIPLite 실전 분석서`
4. `C++로 읽는 SIPLite`
5. `SIPLite: Registrar, Proxy, TLS 구현 읽기`

가장 무난한 것은 `SIPLite 코드 분석`이고, 가장 설명적인 것은 `SIPLite: Registrar, Proxy, TLS 구현 읽기`다.

## 29.4 독자층에 따른 편집 방향

이 책의 독자층은 하나가 아니다. 크게 세 부류로 나뉜다.

### 1) SIP 입문자

이 독자에게는 용어 설명과 시퀀스 다이어그램이 더 중요하다. 따라서 1부와 2부를 더 부드럽게 다듬고, RFC 인용은 최소화하는 편이 좋다.

### 2) C++ 네트워크 개발자

이 독자에게는 `SipCore`, transport 구조, 큐, TLS 구현, cleanup 정책이 핵심이다. 즉 구조 장과 비판적 읽기 장을 더 두껍게 가져가면 좋다.

### 3) 운영/테스트 담당자

이 독자에게는 배포, 로그, 보안, 체크리스트, 테스트 시나리오가 중요하다. 이 경우 4부와 5부 일부를 앞당겨 배치할 수도 있다.

출판 시 한 권으로 묶는다면, 본문은 1번과 2번 독자층을 기준으로 쓰고 운영·체크리스트는 부록에 가깝게 두는 편이 좋다.

## 29.5 각 부의 권장 톤

책의 일관성을 위해 부마다 톤을 다르게 두는 것이 효과적이다.

- 1부: 설명적
- 2부: 흐름 중심
- 3부: 구현 중심
- 4부: 실용 중심
- 5부: 비판적이되 냉정한 톤
- 6부: 참조 문서 톤

이 차이를 의식하면 책이 단조롭지 않고, 독자도 지금 무엇을 읽고 있는지 쉽게 느낄 수 있다.

## 29.6 장 사이 연결 문장 계획

출판용 원고에서는 장과 장 사이를 이어 주는 문장이 중요하다. 현재 초안은 독립 문서 성격이 강하므로, 다음 연결을 추가하면 읽기 흐름이 좋아진다.

1. 개요 장 끝에서 `main.cpp` 진입을 예고
2. 구조 장 끝에서 REGISTER/INVITE 흐름 장을 예고
3. SIP 흐름 장 끝에서 transport/TLS 장을 예고
4. 구현 장 끝에서 테스트/운영 장을 예고
5. 운영 장 끝에서 리스크와 개선 장을 예고

이 방식은 독자가 한 장을 읽고 난 뒤 자연스럽게 다음 장으로 이동하게 만든다.

## 29.7 축약 또는 통합 가능한 장

출판 분량을 조정하려면 몇 개 장은 통합도 가능하다.

- `07_tests_and_verification.md`와 `25_appendix_test_scenarios.md`
- `15_console_and_operations.md`와 `28_operations_checklist.md`
- `03_transport_layers.md`와 `05_tls_implementation.md`
- `08_risks_and_next_work.md`와 `23_architecture_refactoring_roadmap.md`

반대로 독자를 세분화하려면 각 장을 더 쪼개도 된다. 예를 들어 Digest 인증, TLS, XML 설정은 독립 미니 챕터로 충분히 확장 가능하다.

## 29.8 이 장의 핵심 정리

지금까지 만든 원고는 이미 분석서로는 충분히 의미 있다. 출판용으로 만들기 위해 필요한 것은 새로운 내용보다 "재배치"와 "연결"이다.

즉 다음 편집 단계의 핵심은 세 가지다.

1. 장을 부 단위로 재구성한다.
2. 독자층을 기준으로 설명 밀도를 조정한다.
3. 독립 문서를 책의 흐름으로 다시 연결한다.

실제 부/장 배열 초안은 [33_publication_toc_draft.md](/home/windmorning/projects/SIPWorks/SIPLite/docs/book/33_publication_toc_draft.md)를 기준 문서로 삼는 편이 가장 관리하기 쉽다.
