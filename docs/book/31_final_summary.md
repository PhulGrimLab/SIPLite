# 31장. 최종 요약

이 장은 지금까지의 분석을 한 번에 정리하는 마지막 장이다. 목적은 세부 기능을 다시 나열하는 것이 아니라, SIPLite가 어떤 프로젝트이며 어떤 강점과 한계를 갖는지 한눈에 보이게 만드는 것이다.

## 31.1 SIPLite를 한 문장으로 정의하면

현재 코드 기준으로 SIPLite는 다음처럼 정의할 수 있다.

`SIPLite는 하나의 SipCore를 중심으로 UDP, TCP, TLS transport를 공유하며, registrar와 stateful proxy, subscription 처리 기능을 함께 수행하는 경량 SIP 서버다.`

이 정의에는 이 프로젝트의 핵심이 거의 모두 들어 있다.

- 단순 파서가 아니다.
- 단순 transport 데모도 아니다.
- 단순 registrar도 아니다.
- 하나의 공통 상태 코어를 가진다.

## 31.2 구조적으로 가장 중요한 사실

이 프로젝트를 이해할 때 반드시 기억해야 할 것은 네 가지다.

1. 중심은 `SipCore`다.
2. transport는 UDP/TCP/TLS로 분리되지만 상태 코어는 공통이다.
3. REGISTER, INVITE, ACK, BYE, CANCEL, SUBSCRIBE, NOTIFY가 모두 하나의 코어에서 처리된다.
4. `main.cpp`는 단순 진입점이 아니라 전체 런타임을 조립하는 역할을 한다.

즉 파일 수는 여러 개지만, 실제 구조는 매우 강한 중심축을 가진다.

## 31.3 기능적으로 어디까지 와 있는가

현재 구현된 범위를 정리하면 다음과 같다.

- REGISTER와 Digest 인증
- INVITE 라우팅과 응답 전달
- ACK, BYE, CANCEL 처리
- SUBSCRIBE/NOTIFY 처리
- Timer C 기반 INVITE timeout 정리
- UDP/TCP/TLS transport 지원
- XML 기반 정적 단말 bootstrap
- 로그, 콘솔, 테스트, cleanup 루프

특히 TLS는 흔적만 있는 기능이 아니라, OpenSSL과 실제 연결 관리, 인증서 로딩, transport-aware SIP 헤더 구성을 포함한 실구현이다.

## 31.4 이 프로젝트의 강점

분석 결과 기준으로 강점은 분명하다.

### 구조적 강점

- transport와 SIP 의미 처리를 분리했다.
- 공통 `SipCore`를 통해 기능 일관성을 유지한다.
- 멀티워커와 `Call-ID` 기반 분산 구조를 갖는다.

### 기능적 강점

- REGISTER와 INVITE 계열 흐름이 실제 상태 모델과 연결되어 있다.
- SUBSCRIBE/NOTIFY, Timer C, TLS 등 확장 요소가 이미 들어 있다.
- XML 설정, 로그, 콘솔, 테스트가 함께 있어 실험과 분석이 쉽다.

### 문서화 가치

- 각 기능이 코드 위치와 비교적 직접적으로 연결된다.
- 테스트가 구현 의도를 뒷받침한다.
- 교육용, 분석용, 리팩터링 출발점으로 모두 가치가 있다.

## 31.5 이 프로젝트의 한계

반대로 한계도 분명하다.

### 구조적 한계

- `SipCore`에 책임이 많이 몰려 있다.
- `main.cpp`도 런타임 조립 책임이 커지고 있다.
- 상태 저장소와 처리기가 충분히 분리되어 있지 않다.

### 보안 한계

- TLS hostname verification은 아직 미구현이다.
- self-signed 인증서 자동 생성은 개발에는 좋지만 운영 기본값으로 쓰기 어렵다.
- credential 저장 정책은 단순하다.

### 확장성 한계

- logger 비용과 cleanup 스캔 비용이 커질 수 있다.
- 워커 수 상한이 보수적이다.
- 대규모 production-hardening까지 끝난 구조로 보기는 어렵다.

즉 이 프로젝트는 "좋은 기반"이지 "모든 면에서 완성된 상용 SIP 플랫폼"은 아니다.

## 31.6 이 분석서가 남긴 것

이번 문서화 작업을 통해 남은 것은 단순 요약이 아니다. 실제로는 다음 세 가지 자산이 만들어졌다.

1. 코드 구조를 설명하는 책 초안
2. 운영과 테스트를 위한 참조 문서
3. 리팩터링과 하드닝을 위한 분석 기반

이 세 가지는 서로 다른 독자층에게 각각 의미가 있다.

- 개발자는 구조와 흐름을 따라갈 수 있다.
- 운영자는 체크리스트와 로그 장을 활용할 수 있다.
- 저자는 이 원고를 바탕으로 출판용 책으로 확장할 수 있다.

## 31.7 앞으로 이어질 수 있는 작업

이 문서는 끝이 아니라 기반이다. 다음 작업은 크게 세 갈래로 이어질 수 있다.

### 1) 출판용 편집

- 그림과 표 추가
- 중복 장 정리
- 장 사이 연결 문장 보강
- 용어 설명 다듬기

### 2) 코드 개선

- 상태 저장소 분리
- 메서드 처리기 분리
- TLS hostname verification 구현
- 테스트 강화

### 3) 실험과 검증

- 성능 측정
- TLS 통합 테스트
- 장애 복구 시나리오 정리
- 운영 환경 검증

즉 이 분석서는 코드 읽기의 종착점이 아니라, 다음 작업을 더 빠르게 시작하기 위한 지도다.

## 31.8 마지막 결론

SIPLite는 작은 프로젝트처럼 보이지만, 실제로는 SIP 서버가 가져야 할 여러 핵심 요소를 상당히 밀도 있게 담고 있다. registrar, proxy, subscription, TLS, 설정, 테스트, 운영 경로가 모두 한 코드베이스 안에 연결되어 있기 때문이다.

이 프로젝트의 가장 큰 장점은 "배울 수 있는 구조"라는 점이다. 모든 것이 완벽하게 분리된 시스템은 아니지만, 그 덕분에 오히려 한눈에 따라갈 수 있고, 실제 개선 포인트도 선명하게 드러난다.

따라서 이 책의 최종 결론은 단순하다.

`SIPLite는 읽을 가치가 있는 코드베이스이며, 정리할 가치가 있는 프로젝트다.`

이 문장을 기준으로 삼으면, 앞으로의 집필과 리팩터링, 테스트, 운영 문서화도 같은 방향으로 계속 확장할 수 있다.
